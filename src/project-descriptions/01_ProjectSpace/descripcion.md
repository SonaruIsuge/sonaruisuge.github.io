# Project Space

## 概要

Project Space は、インターン期間中にデザイナー担当のインターンと2人で約2か月かけて開発したローカル協力型パーティーゲームです。

最大4人までの同時プレイに対応しており、プレイヤーは宇宙空間でゴミを回収するペンギンとなって、制限時間内にステージ上のゴミを回収します。プレイヤーは無重力空間を自由に移動・浮遊しながら、伸縮アームやシャベルを使い分け、大型ゴミや小型ゴミを適切な処理施設へ運搬します。

--- 
<br>

このプロジェクトは、企画立案から技術検証、設計、実装まで一連の開発工程を経験した初めてのプロジェクトでした。学生時代のようにすぐ実装へ進むのではなく、ドキュメント作成や技術設計、工数見積もりを経て開発を進めるプロセスを学び、ゲーム開発全体を俯瞰して考える視点を身につけることができました。

## 技術的な取り組み

### GDD・TDD・工数見積もり

開発開始直後の1週間は実装を行わず、まず Game Design Document (GDD) を作成しました。ゲーム全体の仕様やルール、システム構成を整理した後、私はプログラム担当として Technical Design Document (TDD) を作成し、各機能の実装方法や技術的な課題を分析しました。

さらに、それぞれのシステムについて工数見積もりを作成し、開発スケジュールを設計しました。これらのドキュメントは、2か月間の開発における指針となり、日々のタスク管理や開発効率の向上に大きく役立ちました。

### Unity Input System を利用したローカルマルチプレイヤー

ローカルマルチプレイでは、「どの入力デバイスがどのプレイヤーを操作するか」を管理する仕組みが必要でした。

調査の結果、Unity Input System を採用し、デバイスのペアリングイベントを利用してプレイヤーとの紐付けを実装しました。また、コントローラー切断時の再接続にも対応し、ゲームを中断することなく入力を復旧できる仕組みを構築しました。

### チュートリアルシステム

プレイテストの結果、操作方法が分かりにくいことが判明したため、短期間でチュートリアルシステムを実装しました。

必要な要素を：**ダイアログ**、**入力制御**、**UI更新**の3つに整理し、async/await を用いてダイアログ進行を制御しました。また、入力状態を Dictionary で管理し、イベント駆動によって各タイミングで入力の有効・無効を切り替えられる仕組みを構築しました。

その結果、数日という短期間でチュートリアル全体を実装し、ゲームへ組み込むことができました。

### コンポーネント化による物理挙動の改善

宇宙空間という設定上、プレイヤーの移動には物理挙動が重要でした。

当初は CharacterController を採用していましたが、調整を重ねても理想的な操作感を得られませんでした。

そこで移動処理をプレイヤークラスから切り離し、Interface を用いて独立したコンポーネントとして設計しました。これにより、他システムへ影響を与えることなく複数の実装を比較・検証できるようになり、最終的には Rigidbody ベースの実装を採用しました。

この設計により、試行錯誤の効率と将来的な拡張性を大きく向上させることができました。

### 世界観と一体化したUI/UX

当初のUIは情報表示と画面遷移のみを目的としていましたが、先輩から「UIもゲーム体験の一部として設計してみよう」という提案を受けました。

そこでロビーUIを宇宙船の操作パネルとしてゲーム世界へ組み込み、Cinemachine、DOTween、非同期処理を組み合わせることで、コントローラー接続やプレイヤー参加の流れを演出として表現しました。

その結果、単なるメニュー操作ではなく、ゲームの世界観へ自然に入り込める体験を実現し、UI/UXを大きく改善することができました。
