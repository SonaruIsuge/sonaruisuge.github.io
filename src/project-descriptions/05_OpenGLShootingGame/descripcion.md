# OpenGL Shooting Game

## 概要

本プロジェクトは、ゲームエンジンを使用せず OpenGL 上で開発した 2D 弾幕シューティングゲームです。ゲームそのものを制作することではなく、レンダリング、入力処理、オブジェクト管理、衝突判定、敵生成、ゲームフローなど、ゲームランタイムを構成する基盤システムを自ら実装し、ゲームエンジンの内部構造を理解することを目的として開発しました。

--- 
<br>

本プロジェクトを通じて、レンダリング・ゲームロジック・リソース管理がどのように連携して動作するのかを低レベルから理解することができました。また、小規模なプロジェクトであっても、将来的な機能追加を見据えた拡張性や保守性を意識してアーキテクチャを設計する重要性を学びました。

## 技術的な取り組み

### ゲームランタイムの自作

> **OpenGL 上でウィンドウ生成から入力・リソース管理・メインループ・描画抽象化まで実装し、ゲームランタイムの仕組みを低レベルから理解。**

本プロジェクトでは、ゲームエンジンが通常提供しているランタイム機能を理解するため、OpenGL上にシンプルなゲームランタイムを構築しました。

ウィンドウ生成、OpenGL コンテキストの初期化、入力登録、リソース読み込み、レンダリング、ゲーム更新、終了処理まで、アプリケーション全体のライフサイクルを自ら実装しています。これにより、各システムがフレームごとにどのような順序で連携して動作するのかを把握できる構成となっています。

メインループでは、入力処理、時間更新、ゲームロジック、衝突判定、敵・弾の更新、描画、バッファスワップを一定の順序で実行しています。更新順序を明確に管理することで、ゲームオブジェクト間の挙動を安定させ、システム間の依存による予期しない問題を防いでいます。

描画処理は VAO・VBO・GLSL Shader を利用した軽量なレンダリング層として実装しました。ゲームロジックから OpenGL API を直接呼び出さず、描画クラスに責務を集約することで、レンダリング実装とゲームロジックを分離しています。この構成により、将来的なレンダラーの拡張や変更にも対応しやすい設計となっています。

### オブジェクト／コンポーネント設計

> **共通 Object と型安全な Component を組み合わせた ECS-lite な構成で、継承階層を抑えつつ機能追加しやすい設計を実現。**

ゲーム全体は、共通のオブジェクトクラスと Component ベースの設計を組み合わせたアーキテクチャで構成しています。

すべてのゲームオブジェクトは共通の Object クラスを継承し、ライフサイクル管理や描画などの基本機能を提供します。一方で、衝突判定や武器など再利用可能な機能については Component として分離し、継承ではなく組み合わせによって振る舞いを構成できるよう設計しました。

ゲームプレイ上の概念である Character、Enemy、Boss などは継承構造を利用していますが、共通化可能な機能は Component 化することで、クラス階層が肥大化しないよう配慮しています。

また、Component の取得にはテンプレートを利用した型安全なインターフェースを採用し、明示的なキャストを必要としない構成としました。ECS ほど大規模ではありませんが、理解しやすさと拡張性のバランスを重視した設計となっています。

この構成により、新しいゲーム機能を追加する際も既存クラスへの変更を最小限に抑えられ、保守性と再利用性の向上につながりました。

### 弾丸プールと衝突判定管理

> **独自 Object Pool で弾を再利用し、Collision Manager で判定を一元管理することで、大量弾幕時のフレーム安定性を向上。**

弾幕ゲームでは短時間に大量の弾が生成・破棄されるため、メモリアロケーションによる負荷が課題となります。

そこで、すべての弾は独自に実装した Object Pool によって管理しています。弾を生成・破棄するのではなく、使用されていないインスタンスを再利用することで、ゲームプレイ中の動的メモリ確保を削減しています。

この方式により、激しい戦闘中でもアロケーションコストを抑え、フレームレートの安定化につながっています。

また、武器システムは Shooting Component として実装し、弾の生成ではなく Object Pool へ発行要求を行う構成としました。これにより、プレイヤー・敵・ボスが同じ発射システムを共有でき、それぞれ異なる攻撃パターンにも柔軟に対応できます。

衝突判定については、軽量な円形コライダーを利用し、専用の Collision Manager によって一元管理しています。各ゲームオブジェクトが個別に判定処理を持たないため、それぞれの責務を明確に分離した構成となっています。

### 敵WaveシステムとボスのState Machine

> **Wave 管理をモジュール化し、ボスAIを State Machine で段階ごとに分離することで、読みやすく拡張しやすい敵進行を構築。**

敵の出現は、Wave 単位で管理されるモジュール化されたシステムとして設計しました。

各 Wave は出現する敵の種類や出現タイミング、進行条件などを保持し、ゲーム全体の進行は Level Manager が制御します。一方で、敵クラスは移動や攻撃など、自身の挙動のみを担当する構成とすることで、責務を明確に分離しています。

敵クラスは共通処理を継承しつつ、それぞれ固有の移動や攻撃ロジックのみを実装しています。この構成により、新しい敵の追加やゲームバランス調整を容易に行えるようになっています。

ボス AI については、複雑な条件分岐を一つの Update 関数へ記述するのではなく、State Machine を採用しました。各フェーズを独立した State として管理することで、移動・攻撃・状態遷移を個別に実装できるため、可読性と保守性を大きく向上させています。

また、新しいフェーズを追加する際にも既存ロジックへの影響を最小限に抑えられる構成となっています。

### データ駆動型ジオメトリパイプライン

> **2Dモデル情報を独自 .gd ファイルへ分離し、再コンパイルなしで見た目を調整できるデータ駆動型パイプラインを構築。**

ジオメトリ情報をソースコードへ直接記述するのではなく、データ駆動型のアセット管理方式を採用しました。

モデルデータは独自形式の .gd ファイルとして管理し、頂点座標・色・当たり判定半径などの情報を保持しています。読み込み時にはこれらのデータを解析し、GPU の頂点バッファへ転送することで描画を行います。

ゲームデータをコードから分離することで、オブジェクトの見た目を変更する際にプログラム全体を再コンパイルする必要がなくなり、コンテンツ制作とゲームロジックを独立して管理できるようになりました。

シンプルな実装ではありますが、ランタイムシステムとゲームデータを分離するという考え方は、大規模なゲーム開発におけるアセットパイプラインにも通じる設計思想となっています。
