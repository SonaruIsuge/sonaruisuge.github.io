# Twin Soul

## 概要

Twin Soul は、日本神話をモチーフにしたリズムアクション型の横スクロールゲームです。プレイヤーは2人の主人公を操作し、音楽のビートに合わせて敵と戦いながら物語を進めていきます。

--- 
<br>

本プロジェクトは他社との共同開発であり、開発フローやアーキテクチャは相手企業が主導していました。そのため、既存システムを短期間で理解しながら、その設計思想に沿って機能を実装することが求められました。

また、リモート環境で他社メンバーと連携するため、技術的な内容を正確に共有し、効率よく協力するためのコミュニケーションも重要な経験となりました。

## 技術的な取り組み

### 大規模アーキテクチャの理解

プロジェクトへ参加して最初に直面した課題は、既存システムの複雑な構造を理解することでした。

相手企業では独自のフレームワークや開発ルールが採用されており、多くの処理が何層にも抽象化されていました。そのため、コードを順番に読むだけでは全体像を把握することが難しく、まずは一つのシステムに絞って解析することから始めました。

私は普段から、システムを理解する際には図を描きながら整理するようにしています。クラス同士の関係やデータの流れ、各コンポーネントの責務を可視化することで、システム全体の構造を短期間で理解できるだけでなく、その後の実装時にも「どこを修正すればよいか」を素早く判断できるようになりました。

この経験を通して、新規開発だけではなく、既存コードベースを理解する能力の重要性を改めて実感しました。

### 植生システム

本プロジェクトでは、レベルデザイナーが大量の植生を効率良く配置できるツールの開発を担当しました。

まず、エディタ上で範囲や各種パラメータを指定することで、複数種類の草をランダムに配置できるカスタムツールを作成しました。また、配置済みの植生についても、個別削除や位置変更などが簡単に行えるようにし、レベル制作の作業効率を向上させました。

ランタイムでは、大量の植生による負荷を考慮し、チャンク単位で管理を行う仕組みを採用しました。プレイヤーの位置に応じて必要なエリアのみを更新し、オブジェクトプールを利用して植生を再利用することで、不要な生成・破棄を抑えています。

しかし、実際のレベル制作では想定を大きく超える数の植生が配置され、Draw Call がボトルネックとなりました。

そこで複数の最適化手法を調査した結果、GPU Instancing（DrawMeshInstanced） を導入することを提案・実装しました。GameObject を大量に生成するのではなく、位置情報などの描画データのみを GPU に送る方式へ変更したことで、Draw Call を大幅に削減し、パフォーマンス問題を解決することができました。

### キーコンフィグシステム

プレイヤーが自由にキー設定を変更できるキーコンフィグ機能の実装も担当しました。

このようなシステムを実装するのは初めてでしたが、Unity Input System の公式ドキュメントを調査したところ、リバインド機能やキー重複検出など、多くの機能が既に提供されていることが分かりました。

そのため、自前で実装するのではなく、既存機能を活用しながら、設定データの保存、UI更新、設定画面の操作フローといった周辺システムの設計・実装に注力しました。

この経験から、「まず既存の仕組みを正しく理解し、それを最大限活用する」という考え方の重要性を学びました。

### ローカライズ画像システム

開発中、このプロジェクトでは複数の言語やプラットフォームに対応した異なるテクスチャが必要となりました。

アセット管理を簡素化するため、私は4つの独立したコンポーネントで構成されるモジュール式のローカライズシステムを設計しました：

- ローカライズ済みテクスチャコンポーネント
- テクスチャマネージャー
- ローカライズデータ構造
- カスタムエディタツール

テクスチャアセットはAddressablesを使用して管理しました。ゲームの実行中はターゲットプラットフォームが変化しないため、プラットフォーム固有のアセットは起動時に1回だけ読み込まれました。

言語切り替えについては、オブザーバーパターンを実装し、言語が変更されるたびに、すべてのローカライズ済みコンポーネントが自動的に更新されるようにしました。

デザイナーのワークフローを改善するため、テクスチャ管理を一元化するカスタムエディタウィンドウも開発しました。これにより、ユーザーは複数のScriptableObjectを個別に編集する代わりに、単一のインターフェースからすべてのローカライズ済みテクスチャ、言語、プラットフォームを管理できるようになりました。

これにより、生産性が大幅に向上し、設定ミスが減少しました。

### 敵AI開発

既存の敵AIフレームワークを利用して、新しい敵キャラクターの実装を担当しました。

しかし、システムに関するドキュメントは存在していなかったため、まず基底クラスや各コンポーネントの役割を一つずつ調査し、それぞれがどのように連携しているかを理解するところから作業を開始しました。

実装を進める中で、複数の基底クラスに重複した処理が存在していることに気付きました。そこで、共通処理を抽出して一つのクラスへまとめることで、重複コードを削減し、保守性と再利用性を向上させました。

このタスクを通じて、多層継承による大規模システムの設計を理解するとともに、既存コードを改善しながら開発を進める経験を積むことができました。

### 開発効率を向上させるエディタツール

ゲームシステムの開発だけでなく、チーム全体の作業効率を改善するためのエディタツールも複数開発しました。

一つ目は、レベル名の変更に伴って大量のアセット名を一括変更できる Batch Rename Tool です。

レベル制作が進むにつれ、シーン名や命名規則の変更が頻繁に発生するようになりました。しかし、関連するPrefabやScriptableObjectなどを手作業で変更するのは時間がかかるだけでなく、ミスも起こりやすい作業でした。

そこで、正規表現と命名規則を利用した一括リネームツールを Custom Editor Window として実装し、数百個規模のアセットでも短時間で安全に変更できるようにしました。

もう一つは、Scene View Camera Follow Tool です。

横スクロールゲームでは、開発中にプレイヤー周辺の挙動を何度も確認する必要があります。そのたびに Scene View を移動させることを煩わしく感じたため、Unity Editor の API を調査し、Scene View がプレイヤーを自動追従する Overlay を実装しました。

用途に応じて Focus、Follow、Focus & Follow の3種類のモードを切り替えられるようにした結果、このツールはプロジェクト内の開発者全員が日常的に利用するツールとなりました。

ゲーム機能だけでなく、開発者体験を改善することも、チーム全体の生産性向上につながるということを実感した、非常に印象に残っている取り組みです。
